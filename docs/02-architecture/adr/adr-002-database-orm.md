# ADR-002: PostgreSQL и Spring Data JPA

**Статус:** Принято  
**Дата:** 2026-04

## Контекст

Требуется персистентное хранение пользователей, кэша призывателей, матчей и избранного с реляционными связями.

## Рассмотренные альтернативы

| Вариант | Плюсы | Минусы | Решение |
|---------|-------|--------|---------|
| **PostgreSQL + JPA** | ACID, Flyway, Data Mapper | Настройка ORM | ✅ Выбрано |
| MongoDB | Гибкая схема JSON | Слабее для M:N User–Summoner | Отклонено |
| H2 in-memory | Быстрый старт | Нет production-подобной среды | Отклонено |
| JDBC без ORM | Полный контроль | Больше boilerplate | Отклонено |

## Решение

- СУБД: PostgreSQL (`lol_companion_db`)
- ORM: Hibernate через Spring Data JPA
- Миграции: Flyway (`backend/src/main/resources/db/migration/`)

## Обоснование

Кэш профилей и матчей требует реляционной модели с внешними ключами. JPA реализует паттерн Data Mapper (Фаулер).

## Последствия

- (+) Версионирование схемы через Flyway
- (+) Identity Map (L1 cache Hibernate)
- (−) N+1 при ленивых связях — контролируется fetch join в сервисах
